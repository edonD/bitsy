## 2. Browser Automation / Scraping Approaches

### 2.1 Selenium and Playwright for Polling Web Interfaces

Browser automation tools like **Selenium** and **Playwright** can theoretically be used to interact with ChatGPT (chat.openai.com), Claude (claude.ai), Gemini (gemini.google.com), and Perplexity (perplexity.ai) web interfaces — typing queries, waiting for responses, and scraping the rendered output.

**How it would work:**
1. Launch a headless (or headed) browser instance
2. Navigate to the chat interface and authenticate (session cookies or login flow)
3. Type a brand monitoring query into the chat input
4. Wait for the response to render
5. Extract the response text from the DOM
6. Parse for brand mentions, sentiment, and citations

**Tools:**
- **Selenium**: Older, broader browser support, larger ecosystem. Python: `selenium` package.
- **Playwright** (recommended over Selenium): Faster, better async support, built-in auto-waiting, supports Chromium/Firefox/WebKit. Python: `playwright` package. Node.js: `@playwright/test`.

Source: [Playwright CAPTCHA Guide (BrowserStack)](https://www.browserstack.com/guide/playwright-captcha)

### 2.2 Challenges

**CAPTCHAs and bot detection:**
- All major LLM web interfaces sit behind **Cloudflare** or similar CDN protection. OpenAI's platform.openai.com returned a Cloudflare challenge page during this research, blocking automated access.
- Playwright is detectable because it reveals bot-like traits: headless user agents, missing browser plugins, WebDriver flags, and unusual timing patterns.
- The `playwright-stealth` plugin can mitigate some detection by spoofing user agents, runtime environments, and WebDriver flags, but it **fails against advanced CAPTCHA technologies like Cloudflare Turnstile**.
- Solving CAPTCHAs requires third-party services (2Captcha, Anti-Captcha) at $1–3 per 1,000 solves, adding cost and latency.

**Session management:**
- Web interfaces use session tokens that expire, requiring periodic re-authentication.
- Multi-account management is fragile — accounts can be flagged and suspended.
- Rate limiting is aggressive on web interfaces (often stricter than API rate limits).

**Response parsing fragility:**
- DOM structures change without notice, breaking scrapers overnight.
- Streaming responses require waiting for completion indicators.
- Chat history context contaminates results (each query builds on previous context).

**Terms of Service violations:**
- ChatGPT, Claude, Gemini, and Perplexity **all prohibit automated access** to their web interfaces in their Terms of Service.
- Bypassing CAPTCHAs may violate legal regulations in some jurisdictions, including the CFAA (US) and Computer Misuse Act (UK).
- Account termination is a real risk — services actively detect and ban automated users.

Sources: [Playwright CAPTCHA Bypass (Oxylabs)](https://oxylabs.io/blog/playwright-bypass-captcha) | [ZenRows Cloudflare Bypass Guide](https://www.zenrows.com/blog/playwright-cloudflare-bypass)

### 2.3 Why API Is Preferred Over Scraping

| Factor | API Approach | Scraping Approach |
|--------|-------------|-------------------|
| **Legality** | Fully authorized | ToS violation risk |
| **Reliability** | 99.9%+ uptime SLAs | Breaks on DOM changes |
| **Rate limits** | Documented, predictable | Undocumented, aggressive |
| **Cost** | Predictable per-token | Unpredictable (CAPTCHAs, proxies) |
| **Response quality** | Structured JSON | Requires HTML parsing |
| **Reproducibility** | Temperature/seed controls | No parameter control |
| **Scalability** | Concurrent requests | One session per browser |
| **Maintenance** | SDK updates | Constant scraper fixes |

**Verdict**: APIs are the only viable approach for production brand monitoring. Scraping is useful only for one-off research or accessing features not yet available via API (e.g., Perplexity's web interface may show different results than the API).

## 3. Polling Frequency and Architecture Patterns

### 3.1 How Often Should You Poll?

The optimal polling frequency depends on what you're measuring and how fast it changes:

| Use Case | Frequency | Rationale |
|----------|-----------|-----------|
| **Baseline brand visibility** | Daily | LLM model weights don't change daily; parametric knowledge is static between model updates. Daily polling captures the statistical variance in sampling. |
| **RAG/search-augmented monitoring** | 4–6x daily | Perplexity and grounded Gemini responses change as web sources update. Monitor morning/afternoon/evening cycles. |
| **Crisis/event monitoring** | Hourly | If a brand crisis hits and search-augmented models pick it up, you need fast detection. |
| **Competitive benchmarking** | Weekly | Aggregate trends are more meaningful than daily noise. |
| **Pre-publication simulation** | On-demand | Run before publishing content changes to predict impact. |

**Industry practice**: Tryscope polls 50x/day across ChatGPT, Claude, Gemini, and Perplexity. This maps to roughly **2 polls/hour during business hours** across 4 models, or ~12 queries per model per day with multiple prompt variations.

### 3.2 Batch vs. Streaming Approaches

**Batch processing** (recommended for brand monitoring):
- Accumulate queries (e.g., all brand-monitoring prompts for the day), submit in bulk
- OpenAI Batch API: **50% cost reduction**, results within 24 hours
- Anthropic Message Batches API: **50% discount**, supports up to 300k output tokens with beta header
- Google Gemini Batch API: **50% discount** on standard pricing
- Ideal for non-time-sensitive monitoring (daily/weekly reports)
- Use webhooks instead of polling for completion — submit batch with a callback URL

**Streaming** (for real-time monitoring):
- Tokens arrive as generated, reducing time-to-first-token
- Useful for dashboards showing live brand mention detection
- All four providers support streaming (`"stream": true`)
- Higher engineering complexity (handling partial responses, reconnection)

**Recommendation for brand monitoring**: Use batch processing for scheduled daily polls (saves 50% on API costs). Use streaming only for real-time crisis detection dashboards.

### 3.3 Queue-Based Architecture for Multi-Model Polling

A production brand monitoring system needs to manage hundreds or thousands of queries across multiple models. The standard architecture pattern:

```
┌─────────────────────────────────────────────────────────┐
│                   QUERY SCHEDULER                        │
│  (Cron / CloudWatch Events / Temporal workflow)          │
│  Generates: N brands × M prompts × P models = queries   │
└────────────────┬────────────────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────────────────┐
│                   MESSAGE QUEUE                           │
│  (SQS / RabbitMQ / Redis Streams / BullMQ)               │
│  Each message: {brand, prompt, model, temperature, ...}  │
└────────────────┬────────────────────────────────────────┘
                 │
        ┌────────┴────────┐────────┐────────┐
        ▼                 ▼        ▼        ▼
┌──────────────┐  ┌──────────┐  ┌────┐  ┌────┐
│ OpenAI Worker│  │ Claude   │  │Gem.│  │Pplx│
│ (rate-aware) │  │ Worker   │  │Wkr │  │Wkr │
└──────┬───────┘  └────┬─────┘  └──┬─┘  └──┬─┘
       │               │           │        │
       ▼               ▼           ▼        ▼
┌─────────────────────────────────────────────────────────┐
│                   RESULTS STORE                           │
│  (PostgreSQL / DynamoDB / BigQuery)                       │
│  Stores: query, model, response, brands_mentioned,       │
│          sentiment, position, citations, timestamp        │
└─────────────────────────────────────────────────────────┘
```

**Key design patterns:**
- **Per-provider workers** with independent rate limiters (token bucket matching each provider's limits)
- **Dead letter queues** for failed requests (separate retry logic per provider)
- **Priority queues** for urgent crisis monitoring queries vs. routine daily polls
- **Backpressure** — workers pull from queue at their rate-limited pace; queue absorbs burst scheduling

**AWS serverless pattern** ([AWS Serverless GenAI Architecture](https://aws.amazon.com/blogs/compute/part-2-serverless-generative-ai-architectural-patterns/)):
- API Gateway (WebSocket) → Lambda → SQS → Lambda (per-provider) → DynamoDB
- SQS buffers messages to prevent overwhelming LLM endpoints
- Lambda concurrency limits match API rate limits

### 3.4 Caching Strategies to Reduce Costs

Caching is critical because many brand monitoring queries are repetitive (same prompt template, different brands):

| Strategy | Cost Savings | Implementation |
|----------|-------------|----------------|
| **Prompt caching (provider-native)** | 50–90% on input tokens | OpenAI cached input: 50% off. Anthropic: only uncached tokens count toward rate limits. Google: context caching at ~10% of input price + storage/hour. |
| **Semantic caching** | Up to 73% total cost reduction | Hash the semantic meaning of queries; if a similar query was run recently, return cached result. Libraries: GPTCache, Redis with vector similarity search. |
| **Response caching with TTL** | 30–50% | Cache full responses with time-to-live. For parametric (non-search) queries, a 24-hour TTL is reasonable since model weights don't change. For search-augmented (Perplexity), use shorter TTLs (1–4 hours). |
| **Prefix caching** | Up to 95% with batching | When millions of requests share common system prompts, pre-populate the KV cache. Combined with model distillation and batch processing, can achieve 95%+ cost reduction vs. synchronous APIs. |
| **Template deduplication** | 20–30% | If monitoring 50 brands with the same prompt template, batch them into a single request where possible (e.g., "Rank these CRM tools: {list}"). |

Source: [Redis LLMOps Guide](https://redis.io/blog/large-language-model-operations-guide/)

### 3.5 Handling API Failures and Retries

**Common failure modes:**
- **429 Too Many Requests**: Rate limit exceeded. Back off exponentially.
- **500/502/503**: Server errors. Retry with jitter.
- **Timeout**: Long-running queries (especially with extended thinking). Set generous timeouts (60–120s).
- **Model overloaded**: Provider is at capacity. Queue for retry.
- **Invalid response**: Malformed JSON, truncated output. Retry once, then flag for review.

**Retry strategy (exponential backoff with jitter):**
```python
import time, random

def call_with_retry(api_fn, max_retries=5, base_delay=1.0):
    for attempt in range(max_retries):
        try:
            return api_fn()
        except RateLimitError:
            delay = base_delay * (2 ** attempt) + random.uniform(0, 1)
            time.sleep(delay)
        except ServerError:
            delay = base_delay * (2 ** attempt) + random.uniform(0, 1)
            time.sleep(delay)
    raise MaxRetriesExceeded()
```

**Circuit breaker pattern**: If a provider fails N times in a row, stop sending requests for a cooldown period. This prevents wasting quota on a degraded endpoint and allows the queue to route to alternative models.

**Checkpointing**: For large batch jobs, checkpoint progress to disk/database after each successful response. On restart, skip already-completed queries.

## 4. Temperature and Reproducibility

### 4.1 Effect of Temperature=0 vs. Higher Temperatures

| Temperature | Behavior | Use Case for Brand Monitoring |
|-------------|----------|-------------------------------|
| **0** | Greedy decoding — always picks highest-probability token. Most deterministic, but **not fully deterministic** due to floating-point arithmetic on parallel hardware, MoE routing, and load balancing. | **Recommended for baseline monitoring** — minimizes variance between runs, making it easier to detect real changes in brand visibility vs. random sampling noise. |
| **0.1–0.3** | Near-deterministic with slight variance. | Good for capturing "soft" recommendation patterns — models may mention a brand slightly differently each time, which reveals confidence levels. |
| **0.7–1.0** | Standard creative sampling. Significant variance between runs. | Useful for diversity testing — "how often does brand X appear across many samples?" This requires more samples for statistical significance. |
| **>1.0** | High randomness, potentially incoherent at extremes. | Not recommended for brand monitoring. |

Sources: [Vincent Schmalbach — Temperature 0 Determinism](https://www.vincentschmalbach.com/does-temperature-0-guarantee-deterministic-llm-outputs/) | [Dylan Castillo — Seed & Temperature](https://dylancastillo.co/posts/seed-temperature-llms.html)

### 4.2 Why Temperature=0 Is Not Fully Deterministic

Even with temperature=0, no major LLM provider guarantees identical outputs across runs:

1. **Mixture-of-Experts (MoE) architectures** — GPT-4o, Gemini, and others use MoE routing, where different expert sub-networks may be selected on different hardware, producing subtly different logits.
2. **Floating-point non-determinism** — Matrix multiplications on GPUs are parallelized; the order of floating-point additions affects results due to rounding. Different GPU hardware (A100 vs. H100) produces different results.
3. **Load balancing** — Requests may be routed to different server replicas running slightly different model snapshots or hardware configurations.
4. **System prompt updates** — Providers periodically update system prompts that wrap user queries, changing behavior without model changes.

**Practical observation**: With temperature=0, responses are typically **~90–95% consistent** across runs (same structure, same brands mentioned, minor wording differences). At temperature=1.0, consistency drops to **~60–70%** for recommendation-style queries.

Source: [OpenAI Community — Temperature 0 Inconsistency](https://community.openai.com/t/why-the-api-output-is-inconsistent-even-after-the-temperature-is-set-to-0/329541) | [Unstract — Deterministic LLM Outputs](https://unstract.com/blog/understanding-why-deterministic-output-from-llms-is-nearly-impossible/)

### 4.3 How Many Samples Are Needed

The number of samples depends on the temperature setting and the precision you need:

| Temperature | Samples Needed | Confidence | Rationale |
|-------------|---------------|------------|-----------|
| **0** | 3–5 per query | High | Most responses will be nearly identical. 3 samples detect hardware/routing variance. 5 provides redundancy. |
| **0.3** | 5–10 per query | Moderate-High | Small variations in wording and ordering. 10 samples give good coverage of the probability distribution. |
| **0.7–1.0** | 15–30 per query | Moderate | Significant variance. Need enough samples to estimate brand mention frequency with reasonable confidence intervals (±10% at 95% confidence requires ~25 samples for a brand appearing ~50% of the time). |

**Statistical formula**: For a brand appearing in P% of responses, the number of samples n needed for a margin of error E at confidence level z is: `n = (z² × P × (1-P)) / E²`. For P=0.5, E=0.1, z=1.96: n = 96. For E=0.15: n = 43. For E=0.2: n = 25.

**Industry practice**: GEO tools like Tryscope appear to use 3–5 samples at low temperature for cost efficiency, focusing on structured comparison rather than probabilistic frequency estimation.

### 4.4 Seed Parameter Support Across Models

| Provider | Seed Parameter | Notes |
|----------|---------------|-------|
| **OpenAI** | Yes (`seed` integer) | Available in Chat Completions. Improves reproducibility when all inputs match (model, prompt, temperature, penalties). OpenAI includes a `system_fingerprint` in responses to detect when backend changes may affect determinism. |
| **Anthropic** | **No** | Claude exposes `temperature` and `top_p`/`top_k` but no seed parameter in the public API. No determinism guarantee even at temperature=0. |
| **Google Gemini** | Partial | Available on Vertex AI (`seed` parameter in generation config). Not consistently available in the free Gemini Developer API. |
| **Perplexity** | **No** | No seed parameter. Additionally, results are inherently non-deterministic because they incorporate live web search results that change over time. |

Source: [OpenAI Community — Seed Parameter](https://community.openai.com/t/question-about-the-use-of-seed-parameter-and-deterministic-outputs/773638)

### 5.3 Cost Optimization Strategy

| Strategy | Savings | Implementation |
|----------|---------|----------------|
| Use cheap models for polling (GPT-4o-mini, Flash-Lite) | 90–95% vs. flagship | Poll with mini/Flash-Lite; use premium models only for detailed analysis on detected changes |
| Batch API | 50% | Batch non-urgent daily polls |
| Prompt caching | 50–90% on input | Reuse system prompts across queries |
| Reduce samples at temp=0 | 60–80% | Use 3 samples at temp=0 instead of 30 at temp=1 |
| Semantic caching | 30–73% | Cache similar queries |
| Smart scheduling | 20–40% | Skip polling when no model update or web content change detected |
